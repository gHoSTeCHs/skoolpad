## Architecture

### Shared Component Contract

Every input component follows the contract established by `mcq-input.tsx`:

```tsx
interface XInputProps {
    responseConfig: XConfig;
    onSubmit: (data: Record<string, unknown>) => void;
    feedback?: { isCorrect: boolean | null; correctAnswer: Record<string, unknown> | null } | null;
    readOnly?: boolean;
    existingAnswer?: Record<string, unknown> | null;
}
```

The Submit Answer button lives inside each individual component. `question-answer-input.tsx` gains a dispatch branch for every new type.

## Tier-by-Tier Component Specs

### Tier 1 — Simple Selectors

**`MultiSelectInput`**
- Checkbox cards (same card design as MCQ — `mcq-input.tsx` pattern)
- Submit enabled when ≥1 option selected
- Feedback: correct options → green, incorrect selections → red, missed correct options → green outline
- Grading: exact set match (no partial credit in Phase 1)
- response_data: `{ selected_labels: string[] }`

**`TrueFalseInput`**
- Two large toggle cards ("True" / "False")
- Optional justification textarea if `requires_justification = true` in config
- Grading: boolean match only (justification stored but not graded)
- response_data: `{ answer: boolean, justification?: string }`

**`NumericEntryInput`**
- Single number input + unit label (from config, read-only display)
- Submit enabled when a numeric value is entered
- Grading: `Math.abs(value - answer) <= (tolerance ?? 0)`
- response_data: `{ value: number, unit?: string }`

**`AssertionReasonInput`**
- Displays assertion and reason as styled quote blocks
- 5 radio option cards (options from `response_config.options`, same structure as MCQ)
- Grading: label match against correct option
- response_data: `{ selected: string }`

### Tier 2 — Drag-and-Drop

**`OrderingInput`**
- Vertical sortable list using `@dnd-kit/sortable`
- Draggable cards with drag handle icon
- Submit enabled immediately (always has an order)
- Feedback: each item shows green (correct position) or red (wrong position) border
- Grading: exact array match vs `correct_order`
- response_data: `{ order: number[] }` (student's sequence of item indices)

**`MatchingInput`**
- Two columns: left items are draggable, right column slots accept drops
- Uses `@dnd-kit/core` with droppable zones
- Once all left items are placed, Submit appears
- Feedback: green/red overlay per pair
- Grading: pair mapping match
- response_data: `{ pairs: Record<string, number> }` (left-index → right-index)

### Tier 3 — Inline Inputs

**`FillBlankInput`**
- Question content split at blank markers (`_____` or `[blank]`)
- Inline `<input>` elements injected at each gap position
- Submit enabled when all blanks have a value
- Grading: ≥80% of blanks match (case-insensitive, checks `correct_answers` array per blank)
- response_data: `{ blanks: Record<string, string> }` (position → answer)

**`ClozeInput`**
- Question content split at gap markers
- Inline `<select>` dropdowns injected at each gap with options from config
- Submit enabled when all gaps selected
- Grading: each gap's selected index matches `correct` index
- response_data: `{ gaps: Record<string, number> }` (position → selected index)

### Tier 4 — Text-Only (Non-Gradable)

All four types: `is_correct = null`. Feedback message: "Answer submitted — teacher grading required."

**`ShortAnswerInput`** — Single `<input type="text">`
**`TheoryInput`** — `<textarea>` (min 3 rows)
**`EssayInput`** — `<textarea>` (min 6 rows) + live word count display
**`CalculationInput`** — Numeric input + unit text field + working `<textarea>`

response_data shapes per brief spec.

### Tier 5 — Complex Visual

**`DiagramLabelInput`**
- When image + coords: image with numbered SVG circle markers at x/y% positions, numbered text inputs listed below
- Fallback (no image or no coords): numbered input list only
- Grading: all labels match (case-insensitive)
- Partial grading: correct if all labels match (per brief spec)
- response_data: `{ labels: Record<string, string> }` (hotspot_N → answer)

**`MatrixMatchingInput`**
- Grid table: left items as rows, right items as columns, checkboxes at intersections
- Student checks which right items match each left item
- Submit enabled immediately
- Grading: exact mapping match per row
- response_data: `{ matches: Record<string, number[]> }` (left-index → right-index array)

### Tier 6 — Group Renderer

**`GroupRenderer`**
- No answer input of its own
- Renders children array recursively using `QuestionAnswerInput`
- Each child numbered within the group
- Each child question treated as independent answer in the session

### `PracticeService::gradeAnswer()` — new private methods

| Type | Method | Logic |
|---|---|---|
| `multi_select_mcq` | `gradeMultiSelect` | Exact set match of selected_labels vs correct options |
| `true_false` | `gradeTrueFalse` | Boolean match vs `correct_answer` |
| `cloze` | `gradeCloze` | All gap indices match |
| `matching` | `gradeMatching` | All pair mappings match |
| `ordering` | `gradeOrdering` | Exact order array match |
| `numeric_entry` | `gradeNumericEntry` | Within tolerance |
| `assertion_reason` | `gradeAssertionReason` | Label match (reuses MCQ logic) |
| `matrix_matching` | `gradeMatrixMatching` | All row mappings match |

Returns structured correct answer data per type for display in results page.

## Decisions

- **Matching input:** Drag-and-drop (not dropdowns) — consistent with ordering DnD, `@dnd-kit` already installed
- **Diagram label:** Image with numbered SVG markers + inputs below (not absolute overlay inputs) — mobile-safe, consistent with card-based patterns
- **No partial credit:** Multi-select requires exact match in Phase 1
- **Fill blank threshold:** ≥80% blanks correct = correct (per brief)
